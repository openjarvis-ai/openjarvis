# OpenJarvis

OpenJarvis is a tool that helps you automate day-to-day tasks. It uses screen recording to identify repetitive workflows, then turns them into automated processes. By reducing time on routine, manual work, OpenJarvis lets you focus on higher-value, creative tasks instead.

---

## Features

### Screen Recording
- Record entire screen, specific window, or browser tab
- Pause / resume / stop controls
- Live recording indicator with timer
- Video preview with playback controls
- Download recording locally as `.webm`
- Upload recording to backend API
- Graceful permission error handling

### Workflow Review & Comments
- Browse workflows generated by OpenJarvis and Opus
- View workflow details: title, steps (with status timeline), metadata, tags
- Write text comments about any workflow
- Send comments to Opus via `POST /api/send-to-opus`
- Toast notifications for success / error feedback
- **Only** text comments accepted — no ratings, scores, or annotations

### Dashboard
- Overview stats: recordings, workflows, comments sent, total duration
- Recent activity feed combining recordings and workflows
- Quick-action button to start a new recording

## Setup Instructions

### Prerequisites
- **Node.js** 18.17+ (LTS recommended)
- **npm** 9+ or **pnpm** 8+

### Install & Run

```bash
# 1. Navigate into the project
cd openclaw-studio

# 2. Install dependencies
npm install

# 3. Start the dev server
npm run dev

# 4. Initialize the database (creates tables and seeds mock data)
npm run db:init

# 5. Open in browser
open http://localhost:3000
```

### Database (Turso)

The app uses **Turso** (libSQL) with **Drizzle ORM**. For local development without a Turso account, it defaults to a local SQLite file at `data/local.db`.

**Local development (default):**
- No setup needed. Run `npm run db:init` to create tables and seed data.

**Production with Turso:**
1. [Sign up for Turso](https://turso.tech/) and create a database
2. Add to `.env`:
   ```
   TURSO_DATABASE_URL=libsql://your-database.turso.io
   TURSO_AUTH_TOKEN=your-auth-token
   ```
3. Create tables and seed:
   - `npm run db:init` (uses .env)
   - If data doesn't appear in Turso, run manually:
     ```bash
     npm run db:schema-turso   # creates tables
     npm run db:seed-turso     # seeds data
     ```

### Build for Production

```bash
npm run build
npm start
```

## API Reference

### `GET /api/workflows`
Returns all mock workflows.

**Response:**
```json
{
  "success": true,
  "message": "Workflows fetched successfully",
  "data": [ /* Workflow[] */ ]
}
```

### `POST /api/send-to-opus`
Send a text comment about a workflow to Opus.

**Request body:**
```json
{
  "workflowId": "wf_001",
  "comment": "The workflow missed the step where I opened the dashboard.",
  "recordingId": "rec_456"  // optional
}
```

**Validation rules:**
- `workflowId` — required string
- `comment` — required non-empty string, max 2000 chars
- `recordingId` — optional string
- Any additional fields are **rejected** (no ratings, scores, etc.)

**Response:**
```json
{
  "success": true,
  "message": "Comment sent to Opus successfully.",
  "commentId": "cmt_abc123",
  "timestamp": "2026-02-15T10:30:00Z"
}
```

### `POST /api/upload-recording`
Upload a recorded video file.

**Request:** `multipart/form-data` with a `file` field.

**Response:**
```json
{
  "success": true,
  "message": "Recording uploaded successfully.",
  "recordingId": "rec_xyz789",
  "size": 48500000,
  "timestamp": "2026-02-15T10:30:00Z"
}
```

## Key Design Decisions

1. **Feature-based folder structure** — Components are grouped by feature (`recorder/`, `workflows/`, `dashboard/`) for scalability, not by technical type.

2. **Custom `useScreenRecorder` hook** — All MediaRecorder logic is encapsulated in a single hook with a clean return interface. Components never touch the MediaRecorder API directly.

3. **Context-based toast system** — Lightweight toast notifications via React Context, no external dependency. Toasts auto-dismiss and support success/error/info variants.

4. **Strict comment-only API** — The `/api/send-to-opus` route explicitly rejects any payload fields beyond `workflowId`, `comment`, and `recordingId`. This enforces the product requirement that only text comments are sent.

5. **Design tokens via Tailwind** — Custom `surface-*` and `brand-*` color scales, custom shadows, custom fonts (DM Sans + Instrument Serif + JetBrains Mono) for a premium, non-generic look.

6. **Responsive sidebar** — Collapsible on desktop (72px vs 260px), slide-out drawer on mobile with backdrop overlay.

7. **Progressive UI states** — Every data-dependent view has loading skeletons, empty states, and error handling. No blank screens.

## Future Improvements

- **Dark mode toggle** — The CSS supports `dark:` variants; add a theme switcher in settings
- **Real backend** — Replace mock APIs with a database (Prisma + PostgreSQL) and object storage (S3) for recordings
- **Authentication** — Add NextAuth.js for user sessions
- **Workflow search & filtering** — Add search bar, tag filters, and source filters to the workflow list
- **Recording thumbnails** — Generate video thumbnails using canvas extraction
- **Real-time recording status** — WebSocket-based status updates for collaborative review
- **Keyboard shortcuts** — `Ctrl+Shift+R` to start recording, `Esc` to stop
- **E2E tests** — Playwright tests for critical flows
- **Framer Motion** — Replace CSS animations with Framer Motion for richer page transitions and list animations
